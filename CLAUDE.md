## 🔍 Fable аудит системного промта (2026-07-12) — противоречия и пробелы модулей, нумерация ПА-N

Денис попросил профессиональный аудит модульного системного промта бота (9 файлов `prompt-source/*.md`: `01-persona`, `02-recommendation-rules`, `03-faq`, `04-offtop`, `05-response-format`, `06-catalog`, `07-goal-channel`, `08-examples`, `09-subscription-play`) как единой системы — не по отдельности, а на предмет противоречий и пробелов, видимых только при взгляде на всё сразу. Прочитаны все 9 файлов целиком + `prompt.js` (сборка: `readdirSync().sort()` → алфавитный порядок, кеш шаблона до рестарта, плейсхолдеры `{{CATALOG_GAMES}}`/`{{CATALOG_SUBSCRIPTIONS}}`/`{{CHANNEL}}`/`{{SETUP_PRICE}}`/`{{PRICE_LOOKUP}}`). Правок кода НЕ было — чистый аудит, ничего не исправлено. По итогам методика оформлена как переиспользуемый skill **`poigraem-chatbot-prompt-audit`** (`.claude/skills/poigraem-chatbot-prompt-audit/SKILL.md` в корне репо) — им и пользоваться при следующих ревизиях промта.

**TLDR:** система в целом добротная (видно, что её уже несколько раз чистили — сессии 10/13/17/18), но 5 находок бьют прямо по продажам: у горячего лида («беру, оформляй») нет ни одного разрешённого пути передачи менеджеру; цены подписок захардкожены в правилах/примерах и противоречат друг другу (3000₽ vs 3600₽) и живому прайсу из каталога; few-shot Пример 2 в `08-examples.md` учит поведению, прямо запрещённому правилом «подписка-сначала» из `02`; в режиме подбора по подписке `05` и `09` дают взаимоисключающие указания по полю `inCatalog`, причём серверная подстраховка ОШ-5 (сессия 21, повышение `false→true` по совпадению названия) делает конфликт хуже, а не лечит его; вопрос «не забанят ли аккаунт?» не покрыт ни одним модулем.

**🔴 Критично для продаж:**

- **ПА-1. Пример 2 в `08-examples.md` противоречит правилу «подписка-сначала» из `02`.** `02-recommendation-rules.md:18` требует: игра не в каталоге, но входит в подписку → СНАЧАЛА предложить подписку, escalate price только после отказа. Пример 2 (`08-examples.md:23-35`, «Есть Hogwarts Legacy?») сразу отвечает escalate price без проверки подписок — модели следуют few-shot сильнее правил, это потерянный апсейл на каждом таком диалоге. Фикс: заменить игру в примере на заведомо не-подписочную + явная ремарка в скобках, что её нет ни в каталоге, ни в подписках; добавить 4-й пример «названная игра есть в подписке → предложение подписки без escalate».
- **ПА-2. «Нажми кнопку ниже» в Примере 2 — канало-/конфигурационно-зависимая фраза,** убранная из правил ещё в сессии 17 за эту же причину, но оставшаяся в примере. При `priceLookupAvailable=false` (`prompt.js:35-37`) кнопки может не быть вовсе. Фикс: заменить на нейтральное «могу узнать точную цену — секунду».
- **ПА-3. Поле `inCatalog` в режиме подписки: `09` требует `false`, `05` требует `true`, сервер молча делает `true`.** `09-subscription-play.md:19` — «inCatalog: false» для игр подписки. `05-response-format.md:35` — «ВСЕГДА inCatalog:true», если игра есть в каталоге. Конфликт срабатывает на играх, которые есть и в каталоге продаж, и в подписке (GTA V, RDR2 — по собственным примерам файла). Серверное повышение `false→true` (ОШ-5) гарантирует, что даже при выполнении `09` клиент в режиме «помощника, не продавца» получит карточку с ценой и кнопкой «в корзину» — ровно то давление, которое `09` запрещает. Пример: «Что поиграть в PS Plus Extra?» → бот советует GTA V → в виджете всплывает цена и «в корзину». Фикс: в `09` шаг 3 переписать — если игра есть и в каталоге продаж, ставить `inCatalog:true` (карточка покажет цену), но в тексте `reply` подавать её как игру подписки, цену не называть.
- **ПА-8. Нет пути для готового покупателя.** «Всё, беру, как оформить?» — самый ценный момент воронки — не покрыт нигде: `07` говорит общее «подтолкнуть к оформлению», `03` даёт контакт менеджера только в контексте вопросов (оплата/возврат/техпроблемы/регионы/скидки), а `05:40-41` прямо запрещает escalate manager вне этих тем («Не включай escalate в остальных случаях»). Бот каждый раз импровизирует. Фикс: добавить в `05:40` третий легальный повод для escalate manager — «клиент явно готов купить / просит оформить», плюс утверждённый скрипт в `03` и пример в `08`.
- **ПА-9. «Это легально? Аккаунт не забанят?» — ключевой страх покупателя не покрыт нигде.** Не оплата, не гарантия — отдельная тема, которую модель ответит из общих знаний (либо напугает, либо пообещает лишнего). Фикс: явный триггер в `03` с текстом, утверждённым Денисом, + escalate manager при сомнении.
- **ПА-14. Захардкоженные цены подписок противоречат друг другу и живому прайсу.** `02:45,48-49` и Пример 3 (`08:79`) — «3м = 3000₽»; Пример 3б (`08:65`) — «3600₽»; при этом реальная цена приходит через `{{CATALOG_SUBSCRIPTIONS}}`/`pricesBlock` (`prompt.js:96-103`), и `02:42` прямо говорит «цены подписок знаешь из каталога». Три источника, два из них статичные и уже разошедшиеся — неверная цена в переписке о деньгах бьёт по доверию напрямую. Фикс: везде заменить конкретные суммы на «[цена из раздела ЦЕНЫ ПОДПИСОК]» + строка в `02`, что цифры в примерах — только иллюстрация.

**🟡 Логические:**

- **ПА-4. Пример 3 в `08` ведёт с подписки даже на полу-общий запрос. Фикс: уточнить `02:47` — «клиент сам назвал/явно заинтересовался, а не когда ты её только что предложил на общий запрос».

## Архитектурные решения (нерушимые)

### Два провайдера с runtime-переключением

**Два равноправных канала, не основной/резервный:**

- **OpenRouter** — один ключ ко всем моделям (Anthropic, DeepSeek, Gemini, Qwen). Дефолт для бота.
- **Anthropic Direct** — прямой API, только claude-* модели. Дефолт для магазина-описаний.

**Модель и провайдер живут в таблице `settings` в Supabase, не в .env и не в коде.** Меняются на лету без рестарта. Ключи:

```
chatbot_provider       = 'openrouter' | 'anthropic'
chatbot_model          = id модели в формате провайдера
descriptions_provider  = 'openrouter' | 'anthropic'
descriptions_model     = id модели
```

`.env` содержит только bootstrap-дефолты — если в `settings` ещё нет записей при первом запуске.

**Тогглы — Опция A:** сначала провайдер, потом модель из его списка. Anthropic Direct → только claude-*. OpenRouter → весь зоопарк.

**Двусторонний автофоллбэк** только между Anthropic-каналами:
- OpenRouter упал + модель `anthropic/claude-*` → fallback на Anthropic Direct
- Anthropic Direct упал + claude-* → fallback на OpenRouter `anthropic/claude-*`
- Для не-Anthropic моделей (DeepSeek, Gemini) fallback нет → ошибка llm_unavailable

Маппинг моделей — явный словарь в `llm.js`.

**Не делай:**
- Не хардкодь модель или провайдера в коде — только через `settings`
- Не используй провайдер-специфичные фичи которых нет в OpenAI-совместимом API

**Делай:**
- Стандартный OpenAI SDK с `baseURL: 'https://openrouter.ai/api/v1'` для OpenRouter
- Стандартный Anthropic SDK для Anthropic Direct
- Prompt caching в обоих провайдерах — без него цена х3-х4

### Структурированный ответ от LLM

LLM возвращает JSON:
```json
{
  "reply": "текстовый ответ боту",
  "recommendations": [
    { "title": "...", "inCatalog": true/false, "gameId": "...", "highlight": true }
  ]
}
```

Парсинг с обработкой ошибок — если LLM нарушил формат, деградация: показываем только `reply`.

### Standalone-изоляция

- НЕ модифицируй основной server.js магазина — это отдельный проект
- НЕ создавай зависимости от кода магазина
- Связь с магазином ТОЛЬКО через общую Supabase (read-only для каталога)
